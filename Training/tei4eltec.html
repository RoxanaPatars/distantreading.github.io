TEI4ELTEC : what and why

What is the TEI?

According to Wikipedia, the Text Encoding Initiative (TEI) is a text-centric community of practice in the academic field of digital humanities, operating continuously since the 1980s.

  • An organisation, or an institution ?
  • A ‘club’, a fashion, a religion ?
  • A detailed technical specification ?
  • ... or a means of constructing such specifications ?

See http://books.openedition.org/oep/426/; http://books.openedition.org/oep/1237 (en francais)

More concretely, the Text Encoding Initiative is ...

  • a set of ‘Guidelines’ for the encoding of digital resources, currently using XML
  • an international infrastructure responsible for the maintenance, development, and distribution of those Guidelines
  • an international research community of users of those Guidelines

The TEI provides a standard framework within which different views of a text can be articulated rather than a single perspective.

"TEI" is also short for "Text Encoding for Interchange"

The TEI was created (before the Web!) to make possible

  • The creation, exchange, and integration of digital textual data
    • any kind of text
    • in any language
    • from any culture or epoch
  • The TEI was developed by and for researchers and technicians working in the academic community. It was intended to be useful
    • for beginners, needing to learn well-established and agreed solutions
    • for experts, hoping to create new solutions

Consequently, the TEI provides both general and specific solutions, and its architecture is intended to permit customisation

Where did the TEI come from (and why is it still here)?

  • The TEI foundation myth goes back to 1987, when the US National Endowment for the Humanities financed an international workshop on the possibility of defining ‘text encoding guidelines’
  • This was an early example of what is now commonplace: informal voluntary co-operation between researchers at many different institutions mediated by the internet and by three learned societies (ALLC, ACL, ACH)
  • Major influences (in both directions)
    • libraries and digital archives
    • language engineering
    • editors of literary and historical sources

The TEI's original design goals

  • provide recommendations derived from the existing consensus, where this could be determined
  • prefer general solutions to discipline-specific ones
  • supporting both specialization and extension

The TEI was not designed to provide a complete answer ‘out of the box’

Why was this considered necessary ?

  • The rise and rise of mutually incompatible data formats, hand in hand with the evolution of new technologies !
  • And also perhaps a desire to bring traditional philology up to date

Why is the TEI still here?

  • it codifies the set of concepts of importance to text-focussed scholarly communities -- and reflects their evolution
  • conceived independently of particular technologies, its expression has already migrated from SGML to XML
  • it is owned and maintained by its users
  • it has no obvious competitor (beyond chaos)

The TEI menu

  • The TEI defines several hundred concepts or features (‘significant particularities’) identifiable when reading a text, for example
    • basic structural features such as chapters or paragraphs
    • visually salient features concerned with layout and appearance
    • semantically charged features such as names, dates, measurements
    • paratextual features such as correction, normalisation, deletion
  • a vocabulary and a set of techniques for documenting textual ‘metadata’, analogous to library cataloguing
  • several proposals for representing, embedding or associating different kinds of linguistic analysis

What's not on the TEI menu?

Originally, the TEI ruled out of consideration...

  • the web (it didn't yet exist)
  • text formatting or processing (tex, scribe...)
  • digital images (without transcription)
  • encoding of facts or objects (database country)
  • software

its original focus (metadata, text, textual and linguistic analysis) has now expanded to include all of the above

The current TEI landscape

  • Basic organization of continuous texts
  • Diplomatic transcriptions, images, multimedia, annotations...
  • Formal data : dates, names and named entities (people, places, organizations, objects)
  • Paratextual data, documentary evidence, textual transmission
  • Linguistic analyses of all kinds (including speech and music)
  • Documentation and generation of encoding systems
  • Et cetera: see http://www.tei-c.org/P5/Guidelines/

... in short, a new kind of Encyclopaedia!

The TEI Architecture

  • TEI is a modular system. You use it to create an encoding system that reflects your own needs, by choosing from the TEI's pre-defined modules
  • Each module defines a set of elements and their attributes
  • you can choose just the elements you want, and also (within limits) change their properties
  • you can add in non-TEI elements, either from other standards or completely new

Define your goals clearly before trying to use the TEI !

... leading to the many flavours of TEI

Suppose you want to encode a bibliographic description. The TEI makes you choose between :

  • <bibl> (contains any combination of bibliographic elements and text, or just text)
  • <biblStruct> or <biblFull> (both containing a specific set of bibliographic elements in a specific order and no text)
  • (and other things...)

Is the TEI a standard ?

  • The TEI is not TEI All, or TEI Lite, or TEI Simple, or Epidoc, or CES, or...
  • So what is it?

The TEI was designed to support many approaches

Each box here represents ..

  • a set of elements, their attributes, and constraints about their usage
  • the set of documents which match those constraints
  • TEI All is the set of all TEI definitions/documents
  • A TEI subset is a selection from TEI All
  • A TEI extension permits definitions/documents beyond TEI All

The TEI spirit

A delicate tight rope... or just common sense

  • tagging by consensus
  • using a defined lexicon
  • respect for diversity

Standardization should not mean «do what I do» but rather «explain your decisions in terms I can understand »

What does it mean to be TEI Conformant?

  • be honest: every XML element claiming to belong to the TEI namespace should respect the semantics of its TEI definition
  • be explicit: the formal documentation provided by a TEI ODD makes make explicit how you have chosen to use the TEI
  • The TEI elements in a conformant TEI document must be valid with respect to the TEI All schema

The purpose of these rules is to make ‘blind interchange’ easier... though they cannot guarantee it

Why bother with the TEI ?

  • you could markup your text in HTML 5 or design your own tagging system
  • (but what a waste of time that would be)
  • you could use a database system
  • (but you would need some nontrivial technical support)

Understanding the TEI requires you to master a ‘sweet spot’, midway between technology and the humanities

What does it offer ELTeC?

  • a well-understood conceptual model which is common across the (digital) humanities
  • access to plenty of free tools and experience
  • the ability to share and straightforwardly enrich data for multiple uses

And, in practice...

As noted above, every TEI application begins by deciding on the TEI subset (or extension) it needs to use. For ELTeC, WG1 has identified three customizations:

  • level zero: a minimal set of structural features
  • level one : a slightly richer set of features likely to be automatically detectable in existing digital texts
  • level two : a set of features useful for linguistic analysis, including tokenisation and segmentation.

The goal is to define a guaranteed minimum : if feature x is present in a document, it will always be marked up and in the same way. No optional markup!

Understanding ELTeC: a closer look at the schema

Subject to discussion in WG1 and WG2 !

  • The different ‘levels’ of the ELTeC schema are not really levels in the sense that a text will progress majestically from 0 to 2.
  • It is easy to go from 1 or 2 to 0: a stylesheet simply has to remove some tagging automatically
  • It is also easy (assuming the right software environment) to go from from 0 or 1 to 2: linguistic annotations can be added automatically
  • It is not easy to go from 0 or 2 to 1: human judgment may be required

Why the three levels?

  • texts may come from many sources, some of which already have extensive tagging
  • some (though by no means all) of that tagging is useful for distant reading purposes and we would like to be able to retain it rather than have the expense of adding it. For example:
    • distinguishing verse and prose
    • distinguishing highlighted phrases and their functions (e.g. emphasis, foreign, titles)
    • distinguishing embedded quotations
    • (maybe) distinguishing direct speech from narrative
  • texts which don't make such distinctions are still useful (of course!); the core functions of distant reading (as so far defined) don't require these distinctions.
  • identifying a text as level 1 or as level 0 makes explicit and guarantees a specific level of encoding

The ELTeC Header

Like every TEI header, this has some mandatory components:

  • file description, containing a title statement, a publication statement, and a source description
  • Additionally, some elements which optional in TEI are mandatory in ELTeC
  • And some elements have content models which are more constrained than TEI requires

In general we try to follow the Recommendations for Best Practice in Digital Libraries, but our main goal is to provide easy access to metadata which will be useful for distant reading purposes.

Creating an ELTeC header

Data could be captured in several ways:

  • editing a template file in oXygen
  • using a predefined form to capture the data
  • down-translating from an existing header

Data could be managed/edited/updated :

  • using an external database to hold the data and extracting headers as required
  • along with each text file

Version control is necessary !

Basic template for header

<teiHeader type="novelHeader">  <fileDesc>   <titleStmt>    <title> <!-- title of work -->    </title>    <author> <!-- information about the author -->    </author>   </titleStmt>   <extent> <!-- size of the text, in pages and words -->   </extent>   <publicationStmt> <!-- boilerplate statement about status as part of ELTeC -->   </publicationStmt>   <sourceDesc>    <bibl> <!-- bibliographic description of the printed source -->    </bibl>   </sourceDesc>  </fileDesc>  <profileDesc> <!-- additional descriptive information -->  </profileDesc>  <revisionDesc> <!-- revision information -->  </revisionDesc> </teiHeader>

Components of the profile description

The <profileDesc> of an ELTeC text has two mandatory components:

  • a <langUsage> which lists the different languages used in a text
  • <textClass> which supplies classification information about the text
  • Languages are identified by ISO 646 code, with optional modifiers for region, dialect etc. for example EN-us for American English, EN-gb for British English
  • Text classification is indicated by the <catRef> element, which indicates the selection criteria for the novel
  • Its target supplies pointers to each each category applicable
<profileDesc>  <langUsage>   <language ident="en"/>   <catRef target="#author_m #reprint_3"/>  </langUsage> </profileDesc>

selection and descriptive criteria are defined in the corpus header using the <taxonomy>, <category>, and <catDesc> elements.
